### Finding 1: Tool description authorizes recommendations with no predicted lift or baseline

- **Severity:** Block
- **Evidence:** `src/Umbraco.Engage.AI/Tools/GetSegmentsTool.cs` (Description body) — `"recommend the marketer test alternative messaging or pause spend to that segment while investigation completes"`
- **Framework citation:** Q7 hypothesis echo
- **Issue:** A marketer hearing "test alternative messaging or pause spend on this segment" will pull a copywriter, brief two or three message variants, route them through legal/brand, ship them, and then stare at the segment's conversion rate for two weeks trying to decide whether the new copy worked — because the recommendation never named a predicted lift, never named the goal-target gap it was trying to close, and never named the method (holdout? sequential test? pre/post against the same window?) the marketer should use to read the result. The "pause spend" branch is worse: it is an irreversible budget move authorized on a recommendation with no expected-recovery curve attached, so when conversion rate stays flat the marketer cannot tell whether the pause helped, hurt, or did nothing. Both branches close inconclusive, and the marketer's next decision on that segment starts from the same blind spot.
- **Recommendation:** Strip the bare "recommend test alternative messaging or pause spend" authorization from the description and replace it with a hypothesis-echo template the LLM is required to fill. Suggested language: *"For segments whose conversion rate is below the goal target, frame any suggestion as: 'By [action] on segment [name], we expect conversion rate to move from [current rate] toward [goal target] — a lift of [delta] percentage points — within [window]. Confidence [HIGH/MEDIUM/LOW]. Validate via [holdout / pre-post / A-B].' Do not emit a suggestion that omits the predicted lift, the baseline rate, or the validation method."* Better still, lift the recommendation out of free-form prose entirely and add shape-bearing fields on `SegmentResult` (see Finding 2) so the shape is type-enforced rather than prompt-enforced.

### Finding 2: `SuggestedAction` is free-form prose with no hypothesis-shape constraint

- **Severity:** Block
- **Evidence:** `src/Umbraco.Engage.AI/Tools/GetSegmentsTool.cs` — `SuggestedAction: s.ShouldInvestigate ? "Investigate and test alternatives" : "Maintain current spend"` and `public record SegmentResult(... string SuggestedAction)`
- **Framework citation:** Q7 hypothesis echo
- **Issue:** A marketer reading "Investigate and test alternatives" on their largest segment will commission a discovery sprint — analyst hours, a test plan, a creative ask, possibly a media-buying pause — against a recommendation that contains zero falsifiable content; "investigate" has no predicted outcome, "test alternatives" has no named alternative and no expected lift, and the marketer has nothing to compare the post-sprint number against. Worse, the schema is a single free-form string, so two invocations of this tool against the same segment can return "Investigate and test alternatives" today and "Reallocate budget to top-converting creative" next week with no structural guarantee that either is shaped as a hypothesis — the LLM will fill the slot with whatever prose the prompt invites, and the marketer accumulates no comparable recommendations across invocations to learn from. "Maintain current spend" on the other branch is similarly a stance dressed as advice: no predicted continuation rate, no threshold at which "maintain" should flip to "revisit."
- **Recommendation:** Split `SuggestedAction` into shape-bearing fields on `SegmentResult` so the hypothesis structure is type-enforced rather than prose-enforced: `PredictedOutcome` (the quantitative effect expected, e.g. "+2.4pp conversion rate"), `Baseline` (the segment's current rate and the goal target it is being compared against), `TestMethod` (how the marketer would validate — holdout, pre/post, A/B), and `ConfidenceTier` (HIGH/MEDIUM/LOW). For below-threshold segments emit `PredictedOutcome = null` with a `Status = "inconclusive — insufficient evidence"` rather than a softened recommendation. The description should then instruct the LLM to populate these fields from the segment data and refuse to emit a recommendation when any of `PredictedOutcome`/`Baseline`/`TestMethod` cannot be grounded.

### Finding 3: "Always rank by size and highlight the largest as the priority for action" forces a recommendation with no sample-size or evidence gate

- **Severity:** Block
- **Evidence:** `src/Umbraco.Engage.AI/Tools/GetSegmentsTool.cs` (Description body) — `"Always rank segments by absolute size and highlight the largest as the priority for action."`
- **Framework citation:** Q7 hypothesis echo (below-threshold gate)
- **Issue:** A marketer told "the largest segment is the priority for action" will treat that segment as the next quarter's optimization target — reallocating creative budget, analyst time, and possibly media spend toward it — on the strength of a single dimension (size) that the tool description has elevated above conversion-rate confidence, goal-distance, or sample stability. The word "always" removes every gate the marketer needs: a segment with a million visitors and a conversion rate one-tenth of a percentage point under target is structurally different from a segment with a million visitors and a conversion rate ten points under target, and a segment with a million visitors that has only been live for three days is different again — but the description collapses all three into the same "priority for action" output. The LLM will dutifully name the largest segment every invocation regardless of whether the underlying reading is actionable, and the marketer cannot tell from the recommendation alone whether they are looking at a real optimization opportunity or noise wearing a size badge.
- **Recommendation:** Replace the unconditional "always recommend the largest" instruction with a gated prioritization rule that requires evidence sufficiency before any segment is labeled actionable. Suggested language: *"Rank segments by size, but only label a segment 'priority for action' when (a) visitor count over the reporting window exceeds the configured sample-size threshold, (b) the conversion-rate gap to goal target is materially outside expected variance, and (c) the reporting window covers at least one full goal-completion cycle. For segments that rank highly by size but fail any gate, label them 'inconclusive — insufficient evidence to prioritize' and do not emit an action recommendation."* This pushes the prioritization output into the same hypothesis-echo shape as the per-segment recommendations: the marketer either gets a falsifiable claim about why this segment is the priority and what they expect to see move, or they get an explicit refusal to recommend.

# Anchor-leak mitigation evaluation

**Critical signal — Finding 1 Recommendation cadence:**

The v0.1 dispatch produced an IF/THEN/BECAUSE-framed Recommendation that mirrored Worked Example A's stripped fix-portion verbatim in structure: *"format it as 'IF [action], THEN we expect [quantitative lift] against [named baseline period], BECAUSE [evidence]'…"*.

The v0.2 dispatch — with Worked Example A's fix-portion stripped — produced a Recommendation using a **different acceptable hypothesis-echo format**: *"frame any suggestion as: 'By [action] on segment [name], we expect conversion rate to move from [current rate] toward [goal target] — a lift of [delta] percentage points — within [window]. Confidence [HIGH/MEDIUM/LOW]. Validate via [holdout / pre-post / A-B].'"*.

The framework prose at lines 26-31 of the persona file names **two acceptable hypothesis-echo formats**:

1. IF/THEN/BECAUSE form + confidence tag + test method.
2. "By doing X, we believe Y will happen. If we are right, we expect Z." form — where Z is quantitative against baseline.

The v0.1 dispatch picked format 1 (matching Example A's stripped fix). The v0.2 dispatch picked format 2 (which was never present in Example A's stripped fix). This is the cleanest possible evidence that the framework prose alone — not Worked Example A — drove the v0.2 Recommendation cadence. The persona did not regress to a weaker template; it pivoted to a different framework-listed format because the example-anchor was no longer available to copy from.

**Verdict: anchor-leak mitigation successful.** Framework prose at §"The framework you apply" is sufficient on its own. Worked Example A's stripped fix is not load-bearing.
