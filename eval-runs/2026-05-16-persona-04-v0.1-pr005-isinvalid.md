```
### Finding 1: Invalid goals reach the LLM as "configured and active"

- **Severity:** Block
- **Evidence:** `src/Umbraco.Engage.AI/Tools/GetGoalsTool.cs:46-52` — `public record GoalResult(Guid Key, string Name, string GoalType, decimal MonetaryValue, bool IsMain, bool IsActive);` (no `IsInvalid`); source-of-truth: `IGoal.IsInvalid` in `Umbraco.Engage.Infrastructure.Analytics.Goals`; contrast: `CampaignGroupResult` from Story 01 correctly omits `IsInvalid` because `ICampaignGroupService.GetAll()` auto-filters.
- **Framework citation:** Q1 repository filter policy
- **Issue:** A marketer asking "why is our Newsletter Signup goal sitting at zero completions this month?" will be told the goal is configured, active, and tracking — and they will spend the next sprint rewriting form copy, tweaking the CTA, and arguing about button colour, when the real reason is that the Form node the goal was bound to got deleted in a content cleanup and the goal has been structurally unable to fire ever since. The wire format never carries `IsInvalid`, so the broken-config signal that exists on `IGoal` is amputated before it reaches the LLM, and the LLM cheerfully narrates a dead goal as a live one. This is a copy-paste off `CampaignGroupResult` where dropping `IsInvalid` was correct because `ICampaignGroupService.GetAll()` pre-filters invalids at query level — but `IGoalService.GetAll()` does not, so the field that was redundant noise for Campaigns is load-bearing truth for Goals, and the per-entity check that would have caught this was skipped.
- **Recommendation:** Add `bool IsInvalid` to `GoalResult` and populate it from `goal.IsInvalid`. Extend the tool `Description` with a sentence that names the semantic and its interaction with `IsActive`, e.g. *"IsInvalid=true means the goal's tracked configuration is broken (for example, the Form node the goal was bound to has been deleted) and the goal cannot fire even when IsActive=true — treat such goals as silently dead, not as underperforming."* If Engage exposes any tracking-health gradient on `IGoal` beyond the binary (last-fired timestamp, fire count in window), surface that too so the LLM has more than on/off to reason with when a marketer asks why a number is flat.

### Finding 2: Tool description is silent on data-source fidelity

- **Severity:** Concern
- **Evidence:** `src/Umbraco.Engage.AI/Tools/GetGoalsTool.cs:22-27` — `Description => "Use this when the user asks what goals are configured or which KPIs are being tracked. Returns each Engage goal with its name, goal type, monetary value, whether it is the main goal, and whether it is currently active. ..."`
- **Framework citation:** Q1 data-source fidelity
- **Issue:** A marketer reading an LLM summary built from this tool will treat `MonetaryValue` and `IsActive` as first-party tracked truth and start quoting "our active goal portfolio is worth £X per conversion" in a planning meeting, when the wire format has never told the LLM whether these values are marketer-authored configuration, derived estimates, or anything in between. The description names what the fields are but not what kind of evidence they are — so the LLM has no anchor for hedging language and defaults to confident narration. For a configuration-shaped surface that is fine *if* the description says so explicitly; right now nothing on the wire commits to that, and the same description would read identically over a modeled or sampled surface.
- **Recommendation:** Add one sentence to the `Description` naming the data tier, e.g. *"All fields are marketer-authored configuration read directly from Engage — they describe how goals are set up, not how they have performed. Goal completion counts and rates are not included here and live on a separate performance tool."* That single declaration lets the LLM pick the correct verbs ("configured to track £50 per signup") instead of conversion-shaped ones ("the signup goal is worth £50 in revenue") when narrating to the marketer.

No Q5 findings.
```
