# Positive validation run

**Input:** synthetic `GetSegmentsTool.cs` with three stacked Q7 violations:
1. Tool description authorizes "recommend the marketer test alternative messaging or pause spend" — recommendation without hypothesis-echo shape.
2. `SuggestedAction` result field as free-form `string` — recommendation field without structural hypothesis constraints.
3. "Always rank segments by absolute size and highlight the largest as the priority for action" — forced recommendation without sample-size / effect-size threshold gate.

**Output:**

```
### Finding 1: Description authorizes test/pause recommendations without hypothesis-echo shape

- **Severity:** Block
- **Evidence:** `GetSegmentsTool.cs` (Description property) — `"For segments where the conversion rate is below the configured goal target, recommend the marketer test alternative messaging or pause spend to that segment while investigation completes."`
- **Framework citation:** Q7 hypothesis echo
- **Issue:** A marketer hearing "test alternative messaging or pause spend on this segment" will pull a creative brief into the queue, redirect budget away from a live audience, and watch conversion rates drift over the next two weeks with no way to tell whether the new messaging worked, the pause starved a recovering segment, or the segment was always going to revert to the goal target on its own. The description authorizes two materially different actions — a creative test AND a budget pause — without naming a predicted lift, without naming the baseline the marketer should measure against (last period? rolling 28-day? the goal target itself?), and without naming how to read the result. The marketer commits creative-team hours and ad spend against a recommendation shaped as a suggestion, and the post-mortem has no falsifiable claim to check against.
- **Recommendation:** Strip the "recommend… test alternative messaging or pause spend" language from the description and replace it with a hypothesis-echo gate: *"For segments where conversion rate is below the configured goal target, surface the gap as an observation, not an action. If the tool is to propose a remediation, format it as 'IF [action], THEN we expect [quantitative lift] against [named baseline period], BECAUSE [evidence from the data]', with a confidence tier and an explicit test method (A/B split, holdout, or pre/post window)."* Either remove the recommendation authorization entirely and let this be a data-surface tool, or move the recommendation into shape-bearing result fields (`PredictedLift`, `BaselineWindow`, `TestMethod`, `ConfidenceTier`) so the wire format enforces the shape.

### Finding 2: SuggestedAction is a free-form prose field with no hypothesis structure

- **Severity:** Block
- **Evidence:** `GetSegmentsTool.cs` (SegmentResult record and Execute body) — `SuggestedAction: s.ShouldInvestigate ? "Investigate and test alternatives" : "Maintain current spend"`
- **Framework citation:** Q7 hypothesis echo
- **Issue:** A marketer reading "Investigate and test alternatives" on a 40,000-member segment will commission a discovery round, draft variant creative, and ship a test cycle on a recommendation that named no predicted outcome, no baseline to compare against, and no method for declaring the test conclusive — and the parallel "Maintain current spend" string will quietly authorize inaction on segments that may be silently regressing. The `SuggestedAction` field is a stringly-typed prose slot the LLM can paraphrase, reshape, or soften across invocations, so the same segment asked about twice in the same week can produce two different action recommendations with no shape constraint to anchor them. There is no accumulating learning across invocations because no two recommendations are structured the same way.
- **Recommendation:** Replace the single `SuggestedAction` string with shape-bearing fields on `SegmentResult`: `PredictedOutcome` (the quantitative effect the action is expected to produce, e.g. "+1.5pp conversion rate lift"), `Baseline` (the named comparison reference, e.g. "rolling 28-day segment baseline"), `TestMethod` (split test / holdout / pre-post), and `ConfidenceTier` (HIGH/MEDIUM/LOW). If the tool cannot supply those four fields for a segment, it should omit the recommendation block entirely and return data-only — refusing the recommendation is correct when the shape cannot be filled.

### Finding 3: "Always rank by size and highlight the largest as priority" forces a recommendation without sample-size or evidence gate

- **Severity:** Block
- **Evidence:** `GetSegmentsTool.cs` (Description property) — `"Always rank segments by absolute size and highlight the largest as the priority for action."`
- **Framework citation:** Q7 hypothesis echo (below-threshold gate)
- **Issue:** A marketer told that the largest segment is "the priority for action" will reallocate budget and creative attention toward it regardless of whether that segment's conversion-rate reading is statistically meaningful, whether the gap to the goal target is within noise, or whether the segment is large in headcount but thin in recent conversions — and the word "always" closes the door on the tool ever flagging a reading as inconclusive. A 50,000-visitor segment with 12 conversions has a "conversion rate" that the wire format will dutifully rank and recommend against, even though the marketer cannot falsify any action taken on that signal. The recommendation has no predicted lift attached, no baseline to compare the post-action result against, and no threshold below which the tool refuses to judge.
- **Recommendation:** Remove "always" and replace with a sample-size and effect-size gate: *"For segments with fewer than [N] conversions in the reporting window, or where the gap to the goal target falls inside the segment's historical variance, label the segment 'inconclusive — insufficient signal' and do not produce action recommendations. For segments above threshold, rank by predicted lift × confidence tier, not by raw size, and emit each ranked recommendation in IF/THEN/BECAUSE form with explicit baseline and test method."* Ranking by size alone is a heuristic dressed as a recommendation; the wire format should refuse to recommend before it ranks badly.

Context gap (overlap): Q5 evidence tier may also apply to the `ConfidenceTier` requirement proposed in the Recommendation for Findings 1 and 2 — recommend Data-Trust persona review of how confidence is labeled across the tool surface beyond its appearance inside the hypothesis-echo shape.
```

**Verdict: PASS.** Three Q7 violations correctly identified, framework cited verbatim, voice consequence-first throughout, severity Block defensible on each finding under standalone test (each independently commits the marketer to irreversible resources against unfalsifiable claims).
